# Project AGENTS.md

## 项目定位

这是一个可视化个人创意主页项目。

它同时承担两种角色：

1. 作品集：展示我在 AIGC、内容创作、视觉表达、Codex 小工具、创意策划方向的实际作品与案例。
2. 成长型简历：展示我的能力进度、项目进度、阶段积累、输出频率与方向演化。

这个项目解决的问题是：把“我会什么、做过什么、正在往哪里成长”用一个统一、可展示、可持续更新的主页表达出来。

## 项目目标

### 第一目标

做出一个高级、克制、专业的个人创意主页，让访问者能快速理解：

- 我是谁
- 我当前在做什么
- 我有哪些可展示的作品
- 我的能力进度和项目进度如何变化
- 我未来适合承接什么类型的合作或岗位

### 当前阶段目标

第一版优先完成“可展示、可更新、可扩展”的静态内容结构，不追求复杂系统能力。

当前默认重点：

- 手动维护内容
- 页面先能清晰展示作品与成长轨迹
- 数据先从本地静态文件读取
- 优先用 `data.js` 或后续可替换为 `data.json` 的方式组织数据

暂不做：

- 后台管理
- 登录系统
- 数据库
- CMS
- 复杂权限系统
- 过早工程化

### 版本边界

V1 阶段优先：高级、克制、专业、稳定。

V1 默认不做：

- 游戏化表达
- 复杂后台
- 未经确认的大幅风格放开
- 未经确认的功能边界扩张

如果用户后续明确说明进入 `V2` / `V3` 视觉实验阶段，则可以在保持专业感的前提下，尝试更大胆的视觉表达和交互设计。

但未经用户确认，不要擅自放开风格和功能边界。

## 视觉风格要求

整体方向：高级、克制、专业、清晰、有设计感，但不过度炫技。

### 必须遵守

- 不要游戏化
- 不要“打怪升级”语言
- 不要低幼化视觉隐喻
- 不要夸张动效堆砌
- 不要花哨到影响阅读

### 命名要求

- “经验条”统一叫：`能力进度` / `项目进度`
- 不使用：`升级`、`打怪`、`经验值`、`战力` 等游戏化表达

### 视觉表达原则

- 信息层级清楚，先总览，再细节
- 保持留白，避免界面拥挤
- 排版整洁，突出内容而不是装饰
- 强调作品、案例、过程证据，而不是空泛自我包装
- 动效只服务于阅读节奏和信息引导
- 颜色、字体、间距、组件语言保持统一

### 气质关键词

- 编辑感
- 展览感
- 作品陈列感
- 专业创意人主页
- 冷静、自信、不过度自我营销

## 内容结构原则

页面内容应优先服务以下四类信息：

1. 核心身份与方向
2. 代表作品与案例
3. 能力进度 / 项目进度
4. 当前探索与持续积累

内容优先级判断：

- 优先展示“做过什么”
- 其次展示“正在做什么”
- 再展示“想做什么”

如果新增模块，先判断它是否真的帮助用户理解我的能力、作品或成长轨迹。

不能只是“看起来丰富”，必须有展示价值。

## 开发原则

### 范围控制

本项目默认遵循最小可用修改原则。

- 先把当前最重要的信息展示好
- 不为展示能力而过度重构
- 不在第一版提前建设后台能力
- 不在没有明确收益时增加复杂技术栈

### 数据策略

第一版默认使用手动数据维护。

允许：

- `data.js`
- 后续可平滑迁移的 `data.json`
- 轻量静态结构

不默认引入：

- 后端接口
- 登录
- 数据库
- 在线编辑系统

### 修改规则

任何代理或协作者在修改前应先检查：

1. 这次修改是否直接服务项目目标
2. 是否会破坏“高级、克制、专业”的视觉方向
3. 是否引入了不必要的复杂度
4. 是否可以只做最小改动完成目标

### 文案与交互规则

- 文案要专业、清楚、克制
- 避免热血化、游戏化、喊口号式表达
- 避免把成长路径包装成娱乐化机制
- 交互优先稳定、清晰、易读

## 自测要求

每次仅修改 `AGENTS.md` 或后续项目文件时，执行对应范围的最小自测。

### 本文件相关自测

当任务是维护项目级 `AGENTS.md` 时，至少确认：

1. `AGENTS.md` 已成功创建或更新
2. 文件内容包含：项目目标、视觉风格、开发原则、自测要求
3. 文件中明确写出：
   - 页面风格高级、克制、专业，不游戏化
   - “经验条”命名为“能力进度 / 项目进度”
   - 第一版优先手动数据，不做后台、登录、数据库
4. 本次没有改动其他页面文件

### 页面开发自测基线

未来如果修改页面文件，默认至少检查：

1. 页面是否能正常打开
2. 主要内容是否正常显示
3. 新增模块是否与整体风格一致
4. 命名是否仍然使用“能力进度 / 项目进度”
5. 是否错误引入游戏化表达
6. 是否无必要地新增后台、登录、数据库相关依赖

## 代理执行约束

当 Codex、代理或协作者参与本项目时，默认执行顺序如下：

1. 先理解本文件，再动手
2. 先检查，后修改
3. 先最小改动，再考虑扩展
4. 每次完成后说明：
   - 已完成什么
   - 改了哪些文件
   - 如何测试
   - 是否还有风险

如果用户要求与本文件冲突，以用户当前明确指令为准，但应先提示冲突点。

## 后续使用方式

以后这个文件的作用是：作为本项目的最高级项目说明，约束后续页面设计、内容组织、命名方式、开发范围和自测标准。

任何新任务开始前，默认先参考本文件，再决定是否修改页面。
